O que a API DataJud oferece

Campos disponíveis, tipos de query e capacidades do motor ElasticSearch subjacente

⚖
Contexto: A API DataJud é um índice ElasticSearch do CNJ que indexa dados processuais de todos os tribunais brasileiros. Para o TRT-3 (Tribunal Regional do Trabalho da 3ª Região — Minas Gerais), o índice api_publica_trt3 expõe metadados de processos públicos com granularidade suficiente para análises quantitativas robustas. O endpoint base é https://api-publica.datajud.cnj.jus.br.

Campos Disponíveis

Campo Tipo Uso principal Capacidade
numeroProcesso keyword Identificação única do processo (formato CNJ) filtro · busca
dataAjuizamento date Data de distribuição — formato 20101223000000 — dados corrompidos em ~1% séries temporais · range
classe.nome.keyword keyword Classe processual: Reclamação Trabalhista, ACP, MS… terms agg · filtro

orgaoJulgador.codigo integer Código numérico da vara — mais preciso e mais rápido que o nome filtro preferencial
assuntos.nome.keyword keyword[] Assuntos do processo — array (Horas Extras, FGTS, etc.) terms agg · filtro
movimentos.codigo integer[] Código de movimentação (Tabela Processual Unificada CNJ) pipeline · aninhado
movimentos.nome keyword[] Nome textual do movimento processual terms agg
movimentos.dataHora date[] Timestamp de cada movimentação pipeline · scripted
grau keyword Grau do processo: G1 (1ª instância), G2 (recursal) filtro · terms
formato.nome keyword Eletrônico ou Físico filtro
nivelSigilo integer 0 = público; valores maiores = sigilo parcial/total filtro obrigatório

Capacidades ElasticSearch disponíveis

Aggregations — terms

Contagem e ranking por qualquer campo keyword. Base para ranking de assuntos, classes, varas. Usado extensivamente no dashboard.

por_assunto por_classe por_vara
Aggregations — date_histogram

Séries temporais com granularidade de dia, mês, trimestre ou ano. Detecta sazonalidade e tendências. Atenção: gera buckets futuros vazios — filtrar por doc_count > 0.

calendar_interval fixed_interval
Aggregations — date_range

Faixas de datas — base para distribuição de antiguidade processual. Implementado na aba Eficiência. Não gera buckets vazios — retorna exatamente as faixas definidas.

date_range range numérico
_count endpoint

Endpoint separado para contagem exata sem limitação de 10 mil documentos. Essencial: track_total_hits: true no _search causa timeout.

_count contagem exata
Bool Query — filter context

Combinação de must / should / must_not com term, range, match. O filter context é mais rápido que query context para varas — usa cache do ES e não calcula scoring.

filter context must / must_not
Nested aggregations

Combinar múltiplos níveis. Ex: para cada vara, top-5 assuntos. Usado no comparativo de varas. Mais pesado — usar com filtros restritos.

aggs dentro de aggs nested path

O que descobrimos na prática

Comportamentos reais da API observados durante a construção do dashboard — o que a documentação não diz

⚡
Estas descobertas vieram de tentativa e erro real durante a implementação. Cada item aqui evitou um bug ou um timeout em produção. Documentar para não repetir os erros nas próximas iterações.

CONFIRMADO Delay de indexação: ~47 dias

O último processo indexado na 42ª Vara de BH era de 31/03/2026, acessado em 17/05/2026. Isso representa um gap de ~47 dias entre o ajuizamento/movimentação e a disponibilidade na API. Qualquer análise de "distribuídos no mês" subestima o volume real dos últimos 30–60 dias.

BUG Datas corrompidas — anos 2608–2612

Alguns processos têm dataAjuizamento em formato errado: o padrão deveria ser YYYYMMDD, mas alguns registros têm DDMMYYYY interpretado como ano — gerando datas como 2608 ou 2612. O dashboard filtra explicitamente y <= yearNow + 1 para descartar esses outliers.

TIMEOUT track_total_hits derruba a API

Usar track_total_hits: true no endpoint _search para obter a contagem exata causa timeout consistente. A solução é chamar o endpoint _count separadamente — retorna o total exato sem limitação e sem timeout.

BUG Buckets futuros no date_histogram

O date_histogram com min_doc_count: 0 (padrão) gera buckets para meses futuros com doc_count: 0. Usar .slice(-24) pega os últimos 24 buckets — que podem ser os 24 meses futuros vazios. Solução: filtrar por doc_count > 0 antes do slice.

PERF filter context vs query context

Filtrar por vara usando filter (não must) é significativamente mais rápido — o ES cacheia filtros exatos. Queries que usam query context calculam relevância desnecessariamente para filtros binários como vara ou nivelSigilo.

PERF Queries paralelas causam timeout

Múltiplas queries simultâneas ao DataJud (ex: 5+ em paralelo) sobrecarregam a API e causam timeouts. O servidor usa timeout de 28 segundos (Railway mata conexões após 60s). Queries pesadas como a de varas-comparativo fazem requisições em série quando o grupo é grande.

OBS Busca por número de processo

Usar match tokeniza o número e não casa exato. A busca combina term + query_string + term no número sem formatação com minimum_should_match: 1 para garantir encontrar o processo independente do formato digitado.

OBS orgaoJulgador.nome pode ser lento

Filtrar por orgaoJulgador.nome.keyword funciona mas pode ser lento sem cache warm. A alternativa mais performática é filtrar por orgaoJulgador.codigo (integer), quando o código da vara é conhecido. O dashboard suporta ambos via variáveis de ambiente.

ℹ
O que a API definitivamente não oferece: dados de partes (reclamante/reclamado), valores de causa ou condenação, resultado da sentença (procedente/improcedente/acordo), e tempo em cada fase processual de forma diretamente agregável — este último requereria processar movimentos individualmente no servidor.

🌡

Cache server-side das agregações

As queries de acervo, classes e assuntos raramente mudam em poucos minutos. Cache com TTL de 1h reduziria tempo de carregamento e eliminaria timeouts por uso intensivo.

  • TTL de 1h para acervo, classes, assuntos
  • Cache por vara (chave: nome + tipo de query)
  • Cache em memória (Map) ou Redis se disponível
📍

Concentração geográfica no TRT-3

Comparar BH com varas do interior (Uberlândia, Juiz de Fora, Montes Claros). Identificar quais assuntos dominam em cada região — agronegócio no Triângulo, mineração no Quadrilátero.

  • Filtro por cidade-sede de vara
  • Assuntos predominantes por região
  • Evolução: interior ganha peso a cada ano?
📊

Co-ocorrência de assuntos

Um processo raramente tem um único assunto. Nested aggregation revela quais assuntos aparecem juntos — padrões de violação de direitos em conjunto.

  • Horas Extras + Intervalo Intrajornada
  • Assédio Moral + Verbas Rescisórias
  • Acidente de Trabalho + Dano Moral
🔔

Alerta de processos paralisados

Detectar processos sem movimentação há mais de X dias usando must_not + nested range em movimentos.dataHora. Indicador de acervo represado.

  • Count de processos sem mov. há +180 dias
  • KPI exibível na aba Eficiência
  • Consideração: delay de 47 dias afeta a precisão
📈

Tendências por assunto ao longo dos anos

Nested agg: por assunto → por ano. Mostra se Horas Extras cresceu 40% em 2024 vs 2022. Captura impactos de reformas, decisões do TST e mudanças no mercado de trabalho.

  • Top-5 assuntos × série temporal anual
  • Possível aba "Tendências" com área empilhada
  • Impacto da Reforma Trabalhista de 2017
📤

Exportação CSV / Excel

Botão de export nos gráficos e tabelas para os dados exibidos. Útil para análises offline, relatórios institucionais e compartilhamento com a chefia.

  • Export CSV dos dados de assuntos, classes, volume
  • Export da tabela de processos recentes
  • Sem infraestrutura adicional — client-side via Blob
📸

Histórico de snapshots

Salvar periodicamente o acervo e top assuntos da vara em banco local. Permite ver evolução do acervo mês a mês — dado que a API não guarda histórico de "qual era o acervo em X".

  • Snapshot semanal em banco local (SQLite ou PostgreSQL)
  • Permite gráfico de evolução real do acervo
  • Independente do delay de 47 dias da API
⚡

Detecção de fases processuais

Via movimentos recentes: identificar processos em Instrução, Sentença, Recurso ou Execução. Requer mapear códigos TPU para categorias de fase.

  • Filtro por código de movimento recente
  • Contagem de processos por fase estimada
  • Mais complexo: movimentos são array aninhado

Queries usadas em produção e exemplos adicionais para análises futuras

Query 1 — Acervo total exato (via _count)
Endpoint _count — sem timeout, sem limite de 10k
// POST /api_publica_trt3/_count  ← endpoint separado!
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "orgaoJulgador.nome.keyword": "42ª VARA DO TRABALHO DE BELO HORIZONTE" } }
      ]
    }
  }
}
// → { "count": 1847, "_shards": {...} }
// NUNCA usar track_total_hits: true no _search — causa timeout
Query 2 — Volume mensal (com filtro de datas inválidas)
date_histogram — filtrar doc_count = 0 e anos futuros no servidor
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "orgaoJulgador.nome.keyword": "42ª VARA DO TRABALHO DE BELO HORIZONTE" } }
      ]
    }
  },
  "size": 0,
  "aggs": {
    "por_mes": {
      "date_histogram": {
        "field": "dataAjuizamento",
        "calendar_interval": "month",
        "format": "MM/yyyy"
        // min_doc_count: 0 é o padrão — gera buckets futuros!
      }
    }
  }
}
// No servidor: filtrar buckets com doc_count > 0 E ano <= anosAtual + 1
// Depois: .slice(-36) para pegar os 36 últimos meses com dados reais
// NÃO usar .slice(-24) sem filtrar primeiro — pode pegar só buckets futuros vazios
Query 3 — Distribuição do acervo por antiguidade (date_range)
Aba Eficiência — faixas de tempo sem buckets vazios
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "orgaoJulgador.nome.keyword": "42ª VARA DO TRABALHO DE BELO HORIZONTE" } }
      ]
    }
  },
  "size": 0,
  "aggs": {
    "por_faixa": {
      "date_range": {
        "field": "dataAjuizamento",
        "ranges": [
          { "key": "Menos de 1 ano", "from": "now-1y/d" },
          { "key": "1 a 2 anos",     "from": "now-2y/d", "to": "now-1y/d" },
          { "key": "2 a 4 anos",     "from": "now-4y/d", "to": "now-2y/d" },
          { "key": "Mais de 4 anos", "to":   "now-4y/d" }
        ]
      }
    }
  }
}
// date_range não gera buckets extras — retorna exatamente as faixas definidas
// Vantagem sobre date_histogram para esta análise

Aba Varas — todas as varas do TRT-3 em uma query, depois filtrar BH
{

  "aggs": {
    "varas": {

        "size": 300,

      }
    }
  }
}
// → traz todas as varas do TRT-3 com acervo em uma query

Query 5 — Comparativo temporal (mês ou ano) com top classe
Aba Comparativo — nested agg date_histogram + terms
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "orgaoJulgador.nome.keyword": "42ª VARA DO TRABALHO DE BELO HORIZONTE" } }
      ]
    }
  },
  "size": 0,
  "aggs": {
    "por_periodo": {
      "date_histogram": {
        "field": "dataAjuizamento",
        "calendar_interval": "month",  // ou "year"
        "format": "MM/yyyy",
        "min_doc_count": 0
      },
      "aggs": {
        "por_classe": {
          "terms": { "field": "classe.nome.keyword", "size": 3 }
        }
      }
    }
  }
}
// Filtrar no servidor: doc_count > 0 E ano válido (< anosAtual + 1)
// Usar .filter(b => b.doc_count > 0).slice(-24) — na ordem correta
Query 6 — Busca de processo por número CNJ
match tokeniza — combinar term + query_string para busca confiável
{
  "query": {
    "bool": {
      "should": [
        { "term":         { "numeroProcesso.keyword": "0001234-56.2024.5.03.0042" } },
        { "query_string": { "query": "\"0001234-56.2024.5.03.0042\"", "default_field": "numeroProcesso" } },
        { "term":         { "numeroProcesso.keyword": "0000123456202450300042" } }
      ],
      "minimum_should_match": 1
    }
  },
  "size": 1
}
// 3 strategies: número formatado, query_string com aspas, número só dígitos
// minimum_should_match: 1 — basta um casar
// Não usar match — tokeniza o número e traz resultados errados
Query 7 — Tendências de assuntos ao longo dos anos (futura)
Aggregation aninhada: por assunto → por ano — para análise de evolução
{
  "query": {
    "bool": {
      "filter": [
        { "term": { "orgaoJulgador.nome.keyword": "42ª VARA DO TRABALHO DE BELO HORIZONTE" } },
        { "range": { "dataAjuizamento": { "gte": "now-5y/y" } } }
      ]
    }
  },
  "size": 0,
  "aggs": {
    "top_assuntos": {
      "terms": { "field": "assuntos.nome.keyword", "size": 5 },
      "aggs": {
        "por_ano": {
          "date_histogram": {
            "field": "dataAjuizamento",
            "calendar_interval": "year",
            "format": "yyyy"
          }
        }
      }
    }
  }
}
// → permite ver se "Horas Extras" cresceu 40% em 2024 vs 2022
// Filtrar anos inválidos (>2025) no servidor antes de renderizar

Limitações reais — descobertas em produção

O que a API não oferece e o que descobrimos que funciona diferente do esperado

CRÍTICO Delay de indexação: ~47 dias

Confirmado em maio/2026: o último processo indexado na 42ª Vara era de 31/03/2026. Isso significa que a contagem de "distribuídos no mês" está sempre subestimada para os últimos 30–60 dias. Qualquer análise de "volume recente" precisa dessa ressalva.

CRÍTICO Datas corrompidas (~1% dos processos)

Alguns processos têm dataAjuizamento com dia e mês interpretados como ano — gerando datas como 2608 ou 2612. O dashboard já filtra esses outliers. Qualquer nova query com filtro de datas deve incluir a validação year <= anosAtual + 1.

TÉCNICO track_total_hits causa timeout

Usar track_total_hits: true no endpoint _search para contagem exata causa timeout consistente. Solução: usar o endpoint _count separado — mais rápido e sem limitação de 10k.

TÉCNICO Timeout de 28 segundos

O servidor usa timeout de 28s nas chamadas ao DataJud (Railway mata a conexão HTTP em 60s). Queries muito complexas — especialmente a de varas-comparativo com muitas varas em paralelo — podem estourar esse limite em horários de pico.

Sem dados de partes

Por razões de privacidade, a API pública não retorna nomes de autores, réus, advogados ou empresa empregadora. Não é possível identificar quais empresas são mais processadas ou quais escritórios atuam mais na vara. Confirmado na prática.

Sem dados financeiros ou de resultado

A API não expõe valor da causa, valor da condenação, se houve acordo ou qual parte ganhou. Análises sobre taxa de procedência ou valor médio das condenações são impossíveis apenas com DataJud.

Tempo por fase não é agregável diretamente

Calcular quanto tempo um processo ficou em cada fase (ex: distribuição até sentença) exige buscar os movimentos de cada processo individualmente e calcular no servidor. Não existe aggregation nativa do ES que faça isso.

Rate limiting implícito

Muitas queries simultâneas (5+) causam timeout. A API pública é compartilhada. O dashboard já mitiga isso limitando grupos de varas a 8 e usando Promise.all controlado. Cache server-side seria a solução definitiva.

Sigilo exclui processos da API pública

Processos com nivelSigilo > 0 não aparecem. Em varas com ações envolvendo menores ou saúde mental, as estatísticas podem subestimar o acervo real. Filtrar nivelSigilo: 0 explicitamente garante consistência.

Qualidade do preenchimento varia

A completude dos campos assuntos e movimentos depende do preenchimento correto no PJe. Processos mais antigos tendem a ter menos assuntos cadastrados, distorcendo análises retrospectivas longas.

⚠
Consideração metodológica: o delay de ~47 dias é o dado mais relevante para quem usa o dashboard. Qualquer KPI de "distribuídos este mês" reflete na verdade o mês de cerca de 47 dias atrás. Isso deve estar visível no dashboard — a data do último processo indexado seria um indicador útil.
